# Prompt 编写与审查规范

## 1. Purpose

本文件定义本仓库中 Prompt 的编写规则与审查规则。

它要回答五件事：

1. 什么情况下必须把某段内容视为 Prompt 规则的一部分。
2. Prompt 至少要写清什么，才不至于让模型靠猜。
3. `cand_check` 在什么情况下必须对 Prompt 做充分性审查。
4. Prompt 为了提高 KV cache 命中率，应该如何安排稳定块与动态块。
5. 哪些 Prompt 问题会成为候选收口的阻塞项。

本文件不是某个单独模块的实现说明，而是所有 Prompt-heavy 模块共享的上游规则。

---

## 2. When This Guideline Applies

满足以下任一条件时，本文件默认生效：

1. 模块定义了 Prompt 结构、Prompt 组装顺序或 Prompt Block。
2. 模块定义了 system prompt、role prompt、output prompt、system base 或等价提示词层。
3. 模块行为的正确性高度依赖模型理解：
   - 当前执行者是谁
   - 当前目标是什么
   - 当前状态是什么
   - 当前术语与对象分别指什么
4. 模块需要把运行时上下文、共享规则、状态快照、工具清单、输出协议等内容注入模型输入。

若模块只是“会调用模型”，但并未定义 Prompt 结构、上下文注入链路或 Prompt 审查边界，则本文件可以只作为参考，不自动触发 `cand_check` 的 Prompt Adequacy Review。

---

## 3. Core Principle

Prompt 的核心目标不是“把作者脑中想法大致表达出来”，而是：

1. 让模型只基于当前实际可见输入，就能稳定完成指定任务。
2. 让不同实现者对同一份 Prompt 形成相同理解，而不是各自补猜。
3. 在不破坏可理解性的前提下，尽量提高稳定块复用与 KV cache 命中率。

通俗讲：

1. 作者脑中的默认前提，不算 Prompt 已提供的信息。
2. 只有真正出现在 Prompt、输入协议或已声明注入内容里的信息，才算模型可见上下文。
3. 若模型要做对某件事，还必须依赖未写出的前提、状态事实、对象关系或术语解释，则 Prompt 仍然不够。

---

## 4. Prompt Writing Rules

### 4.1 Role Completeness

Prompt 必须明确写清以下最小角色信息：

1. 当前执行者是谁。
2. 当前轮要完成什么目标。
3. 当前轮不应该做什么。
4. 什么样的输出才算成功完成。

如果这些信息缺失，模型通常会在“解释任务”“扩大职责”或“输出格式”上出现分叉。

若模块同时定义了“模板级角色描述”和“实例级 persona”，还必须额外写清：

1. 模板级角色描述来自哪个字段或哪个稳定块。
2. 实例级 persona 来自哪个字段或哪个稳定块。
3. 两者进入 Prompt 的顺序是什么。
4. 两者若语义冲突，谁优先。

只有角色元数据（例如 ID、目录、模板名）而没有角色定义正文，不算角色信息已经写清。

### 4.2 Context Sufficiency

Prompt 必须提供完成任务所需的最小上下文集，至少要覆盖：

1. 当前处理的对象是什么。
2. 当前对象所处的状态是什么。
3. 当前可使用的输入材料有哪些。
4. 当前约束、边界和禁止事项是什么。
5. 当前输出需要满足的验收口径是什么。

若某项信息是任务正确完成的必要条件，就不能只留在作者脑中。

若 Prompt 同时承载长期角色设定、单轮任务和动态输入，默认还必须把它们拆成三个可区分的层次：

1. 身份层：当前执行者是谁。
2. 任务层：当前回合要完成什么。
3. 上下文层：当前回合看到了什么动态材料。

若这三层混写到无法判断“哪段是在定义身份、哪段是在描述本轮任务、哪段是在提供运行时材料”，默认视为上下文不充分。

### 4.3 Concept Clarity

Prompt 中出现的非共识概念，必须先解释再使用。

这里的“非共识概念”默认包括：

1. 项目自定义术语。
2. 项目自定义状态名。
3. 项目自定义角色语义。
4. 项目自定义对象关系。
5. 项目内特有的流程名、阶段名、协作名词。

只有非常稳定的通用概念，例如 JSON、HTTP、TypeScript interface，这类行业共识术语，才可不逐条解释。

### 4.4 Execution Context Completeness

Prompt 审查不得去猜“作者脑内默认背景”。

可执行的判断标准只有一个：

1. 只基于当前 Prompt、当前输入协议和已声明注入内容，模型是否已经拿到完成任务所需的最小信息集。

若仍需额外假设以下任一内容，默认判定为上下文缺失：

1. 未声明的前提条件。
2. 未注入的运行时状态。
3. 未给出的对象关系。
4. 未解释的术语语义。
5. 未说明的成功标准或输出边界。

若同一份动态事实在 Prompt 中出现多次，还必须写清：

1. 哪个位置是它的正式承载位。
2. 其它位置若再次出现，是摘要、引用还是重复 payload。

默认禁止“同一份用户输入 / 协作消息 / 调度触发事实在两个 Prompt block 中都被当作正式 payload 承载”，因为这会让实现者无法稳定判断最终输入真相。

### 4.5 Output Protocol Clarity

当 Prompt 要求模型输出结构化结果时，输出协议必须写清到可以直接实现解析。

至少要求如下：

1. 必须明确写出结构化结果的正式承载边界，保证实现侧可以稳定截取或定位唯一 payload。
2. 必须明确写出“实现侧先提取哪一段，再对哪一段做解析”，不得把关键解析步骤留给实现者自行猜测。
3. 若当前模块采用 tag 包裹 JSON、代码块 JSON、宿主原生结构化输出或其它专用封装，Prompt 必须把该封装方式写成可直接复制的正式协议，而不是只给口头描述。
4. 若采用 tag 包裹 JSON，Prompt 中建议用两行“原样可复制”的方式给出，例如：
   - `Start Tag: <xxx>`
   - `End Tag: </xxx>`
5. 若采用 tag 包裹 JSON，必须明确 tag 内只承载一段合法 JSON；如需多个对象，必须用数组或字段承载。
6. 禁止把 Markdown 代码块、解释文本、注释或第二段结构化 payload 混入正式承载位。
7. Prompt 末尾建议加一句明确收口指令，例如“现在开始只输出正式承载位中的结构化结果”。

### 4.6 TypeScript Schema Requirement

若 Prompt 要求输出 JSON，必须提供完整 TypeScript schema。

要求如下：

1. 字段名必须和实际输出一致。
2. 字段类型必须和实际输出一致。
3. 必填/选填约束必须写清。
4. 联合类型、枚举值、嵌套对象必须完整展开，不得只写“类似某结构”。

### 4.7 Few-shot Example Requirement

若 Prompt 的输出结构、动作语义或边界判断较复杂，必须提供至少 1 个 few-shot example。

要求如下：

1. 示例必须与 schema 一致。
2. 示例必须可直接作为输出参考。
3. 示例优先覆盖最容易歧义的动作或边界。

### 4.8 Logical Closure

Prompt 中的目标、规则、约束、输出格式之间不得互相冲突。

至少要满足：

1. 先定义对象，再引用对象。
2. 先定义术语，再使用术语。
3. 先说明成功标准，再要求模型做判断。
4. 约束与示例不得互相矛盾。
5. 输出协议不得和验收标准打架。

## 5. KV Cache Friendly Ordering

### 5.1 Goal

在不破坏可理解性和语义闭合的前提下，Prompt 应尽量让稳定不变的内容靠前，以提高 KV cache 命中率。

### 5.2 Preferred Ordering

默认优先顺序如下：

1. 稳定系统规则。
2. 稳定角色定义。
3. 稳定输出协议。
4. 稳定工具或资源说明。
5. 运行时上下文与状态快照。
6. 当前轮动态摘要。
7. 当前用户输入或本轮临时目标。

### 5.3 What Counts As Stable

通常可视为稳定块的内容包括：

1. 长期不变的 system rules。
2. 固定角色职责。
3. 固定输出格式与 tag 协议。
4. 固定的工具说明。
5. 长期稳定的模块边界说明。

通常应视为高波动块的内容包括：

1. 当前用户输入。
2. 当前轮状态摘要。
3. 当前动态注入的 history。
4. 本轮临时 artifacts 或运行事实。

### 5.4 Priority Rule

KV cache 友好排序服从以下固定优先级：

1. 可理解性
2. 正确性
3. cache 友好性

因此：

1. 不得为了 cache 命中，把术语解释放到术语使用之后。
2. 不得为了 cache 命中，把角色定义拆散到多个难以理解的位置。
3. 不得为了 cache 命中，把关键前提挪到模型读取链路的后面。
4. 若固定前置与语义清晰冲突，必须优先语义清晰。

### 5.5 Non-blocking vs Blocking

KV cache 友好排序本身属于正式规则，但默认按以下方式处理：

1. 若 Prompt 只是“还不够 cache 友好”，但不影响理解链路，可记为改进项。
2. 若排序方式已经实质破坏定义顺序、解释顺序或上下文闭合，则应视为 Prompt Adequacy 缺陷，可在 `cand_check` 中阻塞。

## 6. Prompt Adequacy Review

当模块命中本文件第 2 节的适用条件时，`cand_check` 必须执行 Prompt Adequacy Review。

Prompt Adequacy Review 不再按“若干松散维度”执行，而是固定审查三个对象。

### 6.1 基础充分性审查

该审查对象固定覆盖：

1. `Role Completeness`
2. `Context Sufficiency`
3. `Concept Clarity`
4. `Execution Context Completeness`
5. `Logical Closure`

规则：

1. 以上任一项出现关键缺口，都可以形成 `blocked` 或 `fix_required`。
2. 这里的“关键缺口”指会让模型职责、对象语义、上下文闭合或成功标准不再稳定的缺口。

### 6.2 结构化输出审查

仅当 Prompt 要求模型输出结构化结果时，才启用该审查对象。

该审查对象固定覆盖：

1. `Output Protocol Clarity`
2. `TypeScript Schema Requirement`
3. `Few-shot Example Requirement`

规则：

1. 若 Prompt 不要求结构化输出，则该审查对象记为 `n/a`，不得强行阻塞。
2. 若 Prompt 要求结构化输出，则 `Output Protocol Clarity` 与 `TypeScript Schema Requirement` 都属于必审项。
3. `Few-shot Example Requirement` 只在输出结构、动作语义或边界判断较复杂时才是必审项。
4. 当 `Few-shot Example Requirement` 已被触发却未满足时，结论只能是 `blocked` 或 `fix_required`。

### 6.3 排序审查

该审查对象固定对应 `KV Cache Friendly Ordering`。

规则：

1. 若排序只是还不够 cache 友好，但未破坏语义链路，默认只记改进项。
2. 若排序已经打断“先定义、再引用”的理解顺序，则可以形成 `blocked` 或 `fix_required`。

### 6.4 `prompt_adequacy_summary` 最小语义契约

凡是输出或回写 `prompt_adequacy_summary`，至少必须明确写出：

1. 是否命中 Prompt Adequacy Review。
2. `基础充分性审查` 的结论。
3. `结构化输出审查` 是否适用，以及其结论。
4. 若命中结构化输出审查，`Few-shot Example Requirement` 是否被触发，以及是否满足。
5. `排序审查` 的结论。
6. 当前是否存在阻塞项；若无，则明确写 `none`。

补充说明：

1. 该字段不新增独立快照字段，但其正文语义必须满足本节最小契约。
2. 不允许只写“Prompt Adequacy Review passed”这类无法复审的空泛总结。

## 7. Blocking Rules

命中 Prompt Adequacy Review 的模块，默认按以下规则判定：

1. 若角色目标不清，不能稳定判断执行者职责或输出目标，必须阻塞。
2. 若上下文不足，模型无法只依赖当前可见输入完成任务，必须阻塞。
3. 若核心术语未解释，且可能导致多种理解，必须阻塞。
4. 若运行上下文缺失，导致模型必须依赖未声明状态或对象关系，必须阻塞。
5. 若目标、规则、约束、输出格式之间未闭合，导致实现者无法稳定判断 Prompt 是否自洽，必须阻塞。
6. 若 Prompt 要求结构化输出，但输出协议、schema 或必须的 few-shot example 缺失，也必须阻塞。
7. 若 Prompt 排序已经打断语义链路、让前文引用未定义对象或术语，也必须阻塞。
8. 若只是 cache 友好性不佳，但 Prompt 仍然清楚、闭合、可执行，可不阻塞，但应记录改进项。
9. 若模块已定义模板级角色描述或实例级 persona，但 Prompt 未写清来源、顺序或冲突优先级，默认属于 `Role Completeness` 缺口，必须阻塞。
10. 若同一份动态输入在多个 Prompt block 中被重复承载，导致实现者无法判断唯一正式落点，默认属于 `Logical Closure` 与 `Execution Context Completeness` 缺口，必须阻塞或至少 `fix_required`。

## 8. Spec Mapping Requirements

若模块 candidate 定义了 Prompt 注入链路，Spec 至少必须写清：

1. 注入哪些块。
2. 注入顺序是什么。
3. 哪些块是稳定块。
4. 哪些块是动态块。
5. 哪些块可裁剪。
6. 哪些块不可省略。

若模块同时定义模板级角色描述与实例级 persona，Spec 至少必须写清：

1. 模板级角色描述的来源字段。
2. 实例级 persona 的来源字段。
3. 合成顺序是什么。
4. 冲突时谁覆盖谁。
5. 它们进入 Prompt 时是原文投影还是摘要投影。

若模块依赖运行时状态进入 Prompt，Spec 至少必须写清：

1. 最小状态快照字段集是什么。
2. 哪些状态字段是当前版本必须注入的。
3. 哪些状态字段当前版本明确不注入。
4. 超预算时先裁剪哪些状态块。

若模块把当前用户输入、协作消息、调度触发事实或等价动态 payload 注入 Prompt，Spec 至少必须写清：

1. 这类 payload 的唯一正式承载位是什么。
2. 场景块、主 Prompt、输出协议块是否允许重复引用。
3. 若允许重复引用，重复位承载的是摘要还是完整 payload。
4. 哪些块绝对不得重复承载动态 payload。

若模块使用附录、专题展开文件或 Prompt 模板文件承载 Prompt 真相，该附属文件至少必须写清：

1. 稳定角色层。
2. 实例 persona 层（若适用）。
3. 场景层。
4. 动态输入层。
5. 拼装顺序。
6. 可裁剪与不可裁剪块。

若模块需要模型输出结构化结果，Spec 至少必须写清：

1. 输出协议的 tag 规则。
2. 结构化 schema 的约束。
3. few-shot example 是否属于必需组成部分。

## 9. Review Examples

### 9.1 Pass Example

以下场景默认可通过：

1. Prompt 先给出稳定角色与系统规则。
2. 再给出固定输出协议。
3. 再注入当前对象状态、边界和输入材料。
4. 最后给出本轮用户输入。
5. 其中项目特有术语都已先定义再使用。

这类 Prompt 同时满足：

1. 角色清楚。
2. 上下文完整。
3. 术语无歧义。
4. 稳定块前置。

### 9.2 Fail Example

以下场景默认不通过：

1. Prompt 先要求“按 Team 收口规则处理当前共享产物回退请求”。
2. 但没有解释：
   - `Team 收口` 是什么
   - `共享产物回退` 的权限边界是什么
   - 当前执行者是不是 Leader
   - 当前 Team 状态是否已注入
3. 同时为了 cache，把术语解释放到了最后，导致模型先看到命令，再看到定义。

这类 Prompt 同时命中：

1. `Concept Clarity` 缺口。
2. `Execution Context Completeness` 缺口。
3. `KV Cache Friendly Ordering` 破坏理解链路。

---

## 10. Relationship With Other Rules

1. 本文件只定义 Prompt 编写与审查规则，不替代各模块自己的结构化输出协议正文。
2. 若某模块要求模型输出结构化结果，具体输出封装、提取步骤、解析硬规则与失败语义，必须由该模块 Spec 或其正式附属展开文件承接。
3. 若模块命中 `cand_check`，Prompt Adequacy Review 的门禁以 `commands/cand_check.md` 为直接执行入口。
4. 若命令级规则与本文件表述不一致，以命令文件和 `command_policy.md` 的最新联动规则为准，并应在当前任务内修正文档漂移。
