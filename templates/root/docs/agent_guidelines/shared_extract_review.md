# Shared 提取边界审查流程

## 1. Purpose

本流程用于判断某段当前写在模块主文件或模块附属展开文件中的内容，是否已经达到应提取为 Shared Appendix 的标准。

它只回答四件事：

1. 当前候选内容是否已经构成 shared 候选。
2. 若构成，它涉及哪些模块或哪些共享主题。
3. 当前是否应立即提取为 `c_shared_xxx`。
4. 若当前不提取，为什么继续留在模块内。

本流程不是普通模块命令，不属于 `{command}:{module}`，也不进入 `docs/specs/_status.md`。

---

## 2. Scope

本流程默认处理：

1. `shared` 与模块主文件、模块 appendix 之间的边界判定。
2. 用户主动要求的 shared 提取审查。
3. 模块命令执行过程中发现的 shared 候选信号复核。
4. 多模块之间是否已经出现“应只有一份正式定义”的共享真相判断。

本流程默认不处理：

1. 直接创建新的 `c_shared_xxx.md` 或 `s_shared_xxx.md`。
2. 直接修改模块正文来完成 shared 提取。
3. 替代 `cand_check`、`cand_promote`、`shared_flow_reconcile` 执行模块状态推进或状态回退。

---

## 3. Trigger Modes

本流程固定支持两种触发方式。

### 3.1 主动触发

当用户明确表达以下任一意图时，执行者应直接进入本流程：

1. “这段是不是该提成 shared”
2. “审一下要不要抽成公共部分”
3. “这块是不是已经该做共享规则”
4. 其它等价表达

此时允许按当前问题所需范围扩展到多个模块做交叉阅读。

### 3.2 被动触发

当执行者在某个模块命令中发现 shared 候选信号时，可以提示用户是否进入本流程。

固定规则如下：

1. 被动触发默认只允许“提示用户并请求拍板”，不允许自动扩成跨模块 shared 审查。
2. 被动触发默认不阻塞当前模块命令。
3. 只有当当前命令必读范围内已经出现明确双真相源证据时，才允许直接把问题作为当前命令的阻塞 finding 报出，而不等待用户二次确认。

---

## 4. Preconditions

执行前必须确认：

1. 已读取 `docs/agent_guidelines/spec_policy.md` 与 `docs/agent_guidelines/command_policy.md`。
2. 已明确当前候选内容的正式承载位置：
   - 模块主文件
   - 模块 appendix
   - 或当前已存在的 shared 文件附近
3. 已读取当前直接相关模块的当前层主文件；若该模块当前层正文明确引用了 appendix，也必须一并读取。
4. 若当前判断涉及已有 Shared Appendix，必须读取这些 shared 文件。
5. 若当前流程属于主动触发且问题天然涉及多个模块，允许按当前判断所需范围补读其它模块；若属于被动触发，且用户尚未确认扩展范围，则默认不得主动扩到当前模块命令必读范围之外。

补充说明：

1. 本流程的目标是判断“边界是否已成立”，不是先假定 shared 一定要提。
2. 执行者不得因为某段内容“未来可能复用”，就在未满足前提时直接把它视为 shared。

---

## 5. Procedure

执行步骤固定如下：

1. 先定位当前候选内容的正式落点，并确认它当前是写在模块主文件、模块 appendix，还是已有 shared 文件附近。
2. 判断当前候选内容是否命中 shared 候选信号。默认候选信号至少包括：
   - 当前模块正文显式宣称“通用 / 统一 / 多模块共用 / 其它模块复用”
   - 当前模块正在定义高共享倾向对象，例如共享输出协议、共享 fallback、共享对象展开、共享 few-shot、共享失败语义
   - 当前模块内容与已有 shared 或其它正式真相出现明显重名、重职责、重协议语义
3. 若当前只发现候选信号，但尚未确认第二个正式模块也依赖同一条真相，则默认结论只能是：
   - 暂不提取
   - 或提示用户是否需要扩展到跨模块 shared 审查
4. 若用户已主动触发，或已明确允许扩展范围，则继续检查是否已有第二个正式模块命中同一条正式行为真相需求。
5. 判断当前候选内容与其它模块中的相似内容，究竟是：
   - 同一条正式行为真相
   - 还是仅主题相似、实现类似、结构相近
6. 判断标准固定如下：
   - 若这段内容应在仓库里只有一份正式定义，则它属于 shared 候选
   - 若不同模块可以各自演化，只是主题相似或结构相似，则它不属于 shared
7. 若确认已形成 shared 候选，再判断当前是否应立即提取：
   - 若继续分别写在多个模块中，会形成双真相源风险，则应立即提取
   - 若只是刚出现第二个模块需求，但共享边界、命名或主题仍未收口，可先记录为 shared 候选，暂不立即提取
8. 形成结论时，固定输出以下四类之一：
   - 继续留在当前模块，不构成 shared 候选
   - 构成 shared 候选，但当前只记录或提示，不立即提取
   - 构成 shared 候选，建议当前轮提取为 `c_shared_xxx`
   - 已形成双真相源，必须先完成 shared 收口

---

## 6. Decision Criteria

### 6.1 `shared` 的正式定义

`shared` 不是“未来可能复用的内容”，而是：

1. 多个正式模块共同依赖的正式行为真相。
2. 且这条真相应在仓库中只有一份正式定义。

### 6.2 与模块 appendix 的边界

固定边界如下：

1. 第一次出现的内容，默认先留在当前模块主文件或模块 appendix 中。
2. 仅因为“未来可能复用”，不得提前抽成 `shared`。
3. 只有当第二个正式模块出现同一条正式行为真相需求时，才进入 shared 候选判定。
4. `shared` 解决的是“唯一正式定义”问题，不是“主题相似”或“实现类似”问题。

### 6.3 何时应立即提取

若同时满足以下条件，默认应建议当前轮提取为 `c_shared_xxx`：

1. 已出现至少两个正式模块依赖同一条正式行为真相。
2. 若继续把该真相分别写在模块内，会形成双真相源风险。
3. 当前共享主题已经足以被稳定命名和独立承载。

### 6.4 何时必须阻塞

若当前命令必读范围内已经能确认以下任一情况，默认必须阻塞：

1. 当前模块正在重新定义一个已存在 shared 的同一条正式真相。
2. 当前模块与另一个已读取到的正式模块，已经对同一条真相形成双真相源。
3. 若继续保留模块内定义，会直接造成正式语义冲突。

### 6.5 文件模型

`shared` 的文件模型固定如下：

1. 一个共享对象对应一个 shared 文件。
2. 同一仓库可以同时存在多个 shared 文件，例如：
   - `c_shared_xxx.md`
   - `c_shared_yyy.md`
   - `s_shared_xxx.md`
   - `s_shared_yyy.md`
3. 不允许把多个不相干共享主题长期堆进一个总 shared 文件，只因为它们都“属于 shared”。

---

## 7. Output Contract

输出必须至少包含：

1. 当前候选内容的正式落点
2. 当前是否命中 shared 候选信号
3. 当前是否已确认第二个正式模块命中同一条真相
4. 结论类型：
   - 不构成 shared 候选
   - 构成 shared 候选但当前只提示
   - 建议提取为 `c_shared_xxx`
   - 必须先完成 shared 收口
5. 若结论不是“立即提取”，必须明确说明为什么继续留在当前模块
6. 若结论为“建议提取”或“必须先完成 shared 收口”，必须明确涉及的模块或共享主题

---

## 8. Non-Goals

本流程不负责：

1. 自动创建 shared 文件
2. 自动迁移模块正文
3. 自动修改 `_status.md`
4. 自动触发跨模块大范围扫描，而不经过用户确认或当前命令范围授权

---

## 9. Examples

### 9.1 首次出现，不提 shared

`module_a` 第一次定义一段 fallback 协议。  
结论：先留在 `module_a` 当前层正文或 appendix 中，不因未来可能复用就提前提取为 shared。

### 9.2 第二个模块出现同一条真相，进入 shared 候选

`module_b` 也出现与 `module_a` 同一条正式 fallback 协议需求。  
结论：进入 `shared_extract_review`，判断是否应提取为 `c_shared_fallback_xxx.md`。

### 9.3 被动触发只提示

执行 `cand_check:module_b` 时，发现当前正文显式写“多个模块统一复用这套协议”，但当前用户未授权跨模块扩范围。  
结论：提示用户可能存在 shared 候选，并询问是否继续做 shared 提取边界审查；当前命令默认不因该信号直接阻塞。

### 9.4 明确双真相源时阻塞

执行 `cand_check:module_b` 时，当前命令必读范围内已经读到某个现存 shared 正式真相，而 `module_b` 又重新定义了同一条协议。  
结论：当前命令直接报出阻塞问题，要求先完成 shared 收口。
